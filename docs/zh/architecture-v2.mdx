# PaperSkills 新版架构草案

## 定位升级

PaperSkills 不应只被定义为一个“学术 Skill 仓库”，而应被设计为一个面向 AI Agent 的科研工作流系统。

目标不是简单堆叠更多 skill，而是把科研任务拆成一条可触发、可衔接、可复用、可验证的流水线，让 Agent 在不同研究阶段自动进入正确的方法，而不是直接开始生成内容。

新的产品定位建议为：

> PaperSkills is a workflow system for AI-native research, turning vague research intent into tracked literature, structured synthesis, study design, draft writing, and review-driven revision.

对应中文表达：

> PaperSkills 是一个面向 AI Agent 的科研工作流系统，用于将模糊研究想法转化为文献追踪、结构化综述、研究设计、论文写作与基于审稿反馈的修订流程。

## 设计目标

新版架构建议围绕五个目标展开：

1. 让用户按“科研阶段”使用系统，而不是按“文件夹名称”挑 skill。
2. 让每个 skill 都有清晰的上游输入、下游输出和证据边界。
3. 让流程型 skill 与任务型 skill 分层，避免所有 skill 都是平级工具。
4. 让仓库同时适配 Codex、Claude、VS Code Agent 等环境。
5. 让 skill 可测试、可更新、可发布，而不是停留在文档集合。

## 核心工作流

新版 PaperSkills 的默认科研工作流建议如下：

1. `using-paperskills`
   作用：作为系统入口，识别当前任务属于哪个科研阶段，并强制切换到合适的 skill。

2. `research-scoping`
   作用：把模糊想法收敛成研究问题、关键词、范围、目标输出和约束条件。

3. `paper-tracker`
   作用：围绕关键词、作者、期刊、会议或主题做近期论文发现和候选池构建。

4. `literature-review`
   作用：对候选文献进行筛选、聚类、综合，产出 related work 或文献综述初稿。

5. `research-design`
   作用：把文献 gap 转化为研究问题、理论命题、假设、方法、变量、数据与验证路径。

6. `paper-drafting`
   作用：把研究设计和证据台账转成论文大纲、章节职责和证据绑定写作草稿。

7. `peer-review`
   作用：从 reviewer 视角对单篇论文进行批判性评估，输出 major/minor concerns 与 recommendation。

8. `revision-planning`
   作用：将审稿意见、作者回复和现有稿件差距转成具体修订清单和执行顺序。

这一工作流对应的是一条闭环链路：

`研究想法 -> 范围收敛 -> 文献发现 -> 文献综合 -> 研究设计 -> 草稿生成 -> 审稿模拟 -> 修订计划`

## 技能分层

新版架构建议采用三层 skill 体系。

### 1. 系统入口层

- `using-paperskills`

职责：

- 会话开始时判断任务类型。
- 优先触发正确 skill，而不是让 Agent 直接回答。
- 当用户请求过宽时，先收敛问题。
- 当证据不足时，强制标注证据边界与风险。

这一层参考 `superpowers` 的 `using-superpowers`，解决的是“系统如何自动进入正确流程”的问题。

### 2. 流程编排层

- `research-scoping`
- `research-design`
- `paper-drafting`
- `revision-planning`

职责：

- 定义每个阶段的输入输出契约。
- 规定何时调用下游 skill。
- 在阶段之间传递结构化 brief，而不是依赖自由文本上下文。

这一层解决的是“研究流程如何串起来”的问题。

### 3. 任务执行层

- `paper-tracker`
- `literature-review`
- `peer-review`

职责：

- 完成具体的单次任务。
- 输出结构化结果和可复用模板。
- 明确说明证据覆盖范围、来源质量和缺口。

这一层解决的是“单个科研动作如何高质量完成”的问题。

## 结构化输入输出契约

当前 `paperskills` 的单个 skill 已经开始采用 `brief-schema` 思路，这是一个正确方向。新版架构建议把它提升为系统级规范。

每个核心阶段都应有统一的 brief：

1. `scoping-brief`
   包含研究主题、研究问题、边界、关键词、目标交付物、语言、时间范围。

2. `tracking-brief`
   包含主题、查询词、期刊/会议/作者范围、时间窗口、输出数量、输出格式。

3. `review-brief`
   包含综述目标、文献池、筛选标准、主题聚类方式、输出语言、综述深度。

4. `design-brief`
   包含研究 gap、理论视角、变量、假设、方法候选、数据来源、识别策略。

5. `draft-brief`
   包含论文结构、大纲、章节目标、每节证据来源、预期图表和引用风格。

6. `reviewer-brief`
   包含论文文本、目标 venue、论文类型、证据完整度、评审模式。

7. `revision-brief`
   包含审稿意见、作者回应要点、现有稿件问题、修订优先级和截止约束。

核心原则是：

- 上游 skill 的输出，尽量直接成为下游 skill 的输入。
- 所有 brief 都优先 JSON 化，便于脚本、模板和 Agent 复用。
- 文本草稿只是展示层，不应成为唯一状态载体。

## 仓库结构建议

建议将仓库调整为更明确的产品结构：

```text
paperskills/
  README.md
  CHANGELOG.md
  RELEASE-NOTES.md
  docs/
    zh/
    en/
  skills/
    using-paperskills/
    research-scoping/
    paper-tracker/
    literature-review/
    research-design/
    paper-drafting/
    peer-review/
    revision-planning/
    writing-paperskills/
  schemas/
    scoping-brief.schema.json
    tracking-brief.schema.json
    review-brief.schema.json
    design-brief.schema.json
    draft-brief.schema.json
    reviewer-brief.schema.json
    revision-brief.schema.json
  examples/
    workflows/
    briefs/
    outputs/
  tests/
    skill-triggering/
    explicit-skill-requests/
    workflow-smoke/
  related/
```

其中有三个关键点：

1. `schemas/` 从 skill 子目录中抽出来，变成系统共享层。
2. `examples/` 从“每个 skill 各放各的”升级为“跨阶段样例资产库”。
3. `tests/` 直接对标 `superpowers` 的测试思路，验证触发、流程和输出纪律。

## README 重写方向

新版 README 建议参考 `superpowers`，但保持学术产品语境，最少包含以下结构：

1. What is PaperSkills
2. How it works
3. Basic research workflow
4. Current skills
5. Installation
6. Usage examples
7. Contributing
8. Updating
9. Roadmap

README 的核心任务不再是介绍“有哪些 skill”，而是回答：

- 这个系统解决什么问题
- 它默认怎么工作
- 用户从哪里进入
- Agent 为什么会自动调用 skill

## 新增 Skill 建议

优先新增以下 5 个 skill。

### `using-paperskills`

建议 description：

`Use when starting any research-oriented conversation to identify the current research stage and route the agent to the correct PaperSkills workflow before answering directly.`

### `research-scoping`

用途：

- 将模糊课题收敛为研究问题和边界。
- 产出关键词、时间范围、目标文体和后续检索要求。

### `research-design`

用途：

- 从文献 gap 出发形成研究设计。
- 适合论文 proposal、研究计划、method 论证和假设构建。

### `paper-drafting`

用途：

- 从 brief 和证据台账生成论文大纲与章节级草稿。
- 强调 evidence-backed writing，而不是自由生成。

### `revision-planning`

用途：

- 将审稿意见转换为修订任务列表。
- 支持 rebuttal 场景、二轮修改场景和内部预审场景。

## 文档站更新思路

当前站点文档内容偏骨架化，尚未承载产品叙事。建议按两层组织。

### 第一层：面向首次访问者

- 首页：产品定位、默认工作流、当前技能、适用平台
- 快速开始：安装方式、如何触发 skill、示例 prompt
- 架构草案：解释系统如何分层与演进

### 第二层：面向贡献者

- Skill authoring guide
- Brief schema guide
- Testing guide
- Release/update guide

这会比当前 [index.mdx] 和 [getting-started.mdx] 更接近真正可维护的产品文档。

## 测试与验证策略

这是 `superpowers` 最值得借鉴但 `paperskills` 当前缺失最多的部分。

建议建立三类测试：

1. `skill-triggering`
   验证用户请求是否能触发正确 skill。

2. `explicit-skill-requests`
   验证显式点名 skill 时，Agent 是否遵循对应流程。

3. `workflow-smoke`
   给定一条完整研究任务，验证是否能按阶段完成收敛、追踪、综述、设计、审稿或修订。

测试目标不是验证文本“好不好看”，而是验证：

- skill 有没有被调用
- 输入输出是否成型
- 是否遵循证据边界
- 是否进入正确阶段而不是直接生成空泛内容

## 一句话总结

`superpowers` 的真正价值，不在于它有多少 skill，而在于它把 skill 组织成了一个会自动触发、可持续演进、带测试约束的工作流系统。

PaperSkills 的新版设计，也应从“学术技能集合”升级为“AI 原生科研工作流系统”。
