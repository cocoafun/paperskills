它通常放在成稿后期使用，尤其适合投稿前做最后一轮核查；如果正文和参考文献列表还在频繁改动，结果会很快过时。

## 适用场景

- 我想在投稿前做一次参考文献体检
- 我怀疑部分引用有年份、作者或页码错误
- 我想检查是否存在找不到的参考文献
- 我想验证关键 claim 是否被原文真正支持

## 它会做什么

按照当前 skill 设计，这个能力通常会：

1. 从稿件里解析所有引文和参考文献
2. 通过 CrossRef、Semantic Scholar、OpenAlex 等源逐条查找
3. 判断每条文献是 found、partial match 还是 not found
4. 检查作者、年份、标题、页码等元数据是否一致
5. 对有开放获取版本的文献，进一步做 claim verification
6. 输出 HTML 风格的核验报告

## 你最好提供什么输入

- 完整稿件文件
- 如果你只想核查某个章节，也可以说明范围
- 如果要做 claim verification，最好提前说明你愿意接受更高 token 成本

## 常见用法

下面示例统一使用显式 skill 前缀，便于 Agent 稳定触发对应能力。

### 核验整篇论文引用

```text
/cite-verify my-manuscript.md
```

### 指定重点核验 claim

```text
/cite-verify 请检查这篇稿件的全部引用，并特别关注引言和讨论部分的关键论断是否被原文支持。
```

## 输出通常包含什么

- 每条参考文献的存在性检查
- 作者、年份、页码、DOI 是否匹配
- open access 状态
- 可能虚构或高度可疑的引用
- 如果做了深入核验，还会有 claim-level 分析

## 使用时要注意

- 这是逐条核验型任务，成本通常高于一般文本生成
- 没有开放获取全文时，claim verification 可能无法完整进行
- 它适合投稿前质量控制，不适合代替正常写作过程中的基本引文管理

## 推荐搭配流程

通常在 `/peer-review` 之后使用；如果核验中发现关键研究脉络有缺口，可以回到 `/lit-search` 或 `/citation-network` 补检。
